## Issues caught + fixed

### A. Wizard PIN didn't pass through to `/api/mesh/create` (regression in this PR)
**Severity:** medium — silent feature loss; user-entered PIN was discarded.

Browser case 07 expected PIN `"1234"` in the result page; got the auto-generated random PIN. Root cause: the wizard sent `body.pin = state.pin` but the `/api/mesh/create` handler called `_mesh_accounts.create(lion_pubkey, client_ip=client_ip)` without forwarding the `pin` field from the request body.

**Fix:** `focuslock-mail.py` — extract `pin` from request body, validate it's exactly 4 digits, forward to `MeshAccountStore.create(lion_pubkey, pin=req_pin, ...)`. Anything not matching `^\d{4}$` falls through to the auto-generated PIN.

### B. `staging/qa_runner.py` regressed against the audit-C1 mesh-order auth gate (pre-existing)
**Severity:** low (test-only) — but blocked all order-dispatch QA against staging.

`mesh.handle_mesh_order` was hardened in v1.2.0 (audit C1) to require either a valid PIN or a Lion signature whenever the orders doc has `pin` or `lion_pubkey` set. `qa_runner.py` was sending bare `{admin_token, action, params}` and got `200 {"error": "unauthenticated — missing valid pin or signature"}` on every order. Existed since v1.2.0; nobody re-ran qa_runner against a mesh where lion_pubkey had been adopted (it gets adopted on first `/api/mesh/create` call via the `if not _lion_pubkey: _lion_pubkey = ...` branch).

**Fix:** `staging/qa_runner.py` — load `staging/lion_privkey.pem` at startup; sign `canonical({action, params})` with PKCS1v15+SHA256 and attach as `signature` field on every `_admin_order` call.

### C. Relay hardcoded `web_dir = "/opt/focuslock/web"` (developer-experience)
**Severity:** low (DX) — staging couldn't serve `web/` without sudo.

Two sites in `focuslock-mail.py` hardcoded the production install path. Fixed both via env override: `web_dir = os.environ.get("FOCUSLOCK_WEB_DIR", "/opt/focuslock/web")`. Default unchanged in production.

## Findings — UX nits (deferred to audit phase)

These didn't block the QA but should land in the upcoming audit + index restructure:

1. **Step icons read as ASCII placeholders** — `[]` / `[K]` / `[$]` / `[R]` / `[S]` / `[?]` / `[OK]` look like missing-glyph fallbacks. Replace with emoji (📡 🔑 💰 ⚖️ 🪙 ✓ 🎉) or inline SVG.
2. **Welcome bullet list and step labels don't quite match.** Welcome says "Default rules: tribute, bedtime, screen time" — fine, but consider listing exactly what each step does so the user can predict the flow.
3. **Subscription step note about per-tier amounts** — currently says "To customize amounts, change them after setup from Lion's Share." Truth is: there's no UI in Lion's Share to customize the amounts either; per-tier values are server-hardcoded (`{"bronze": 25, "silver": 35, "gold": 50}` in `focuslock-mail.py:792`). The audit should decide whether to add server config or remove the misleading hint.
4. **Auth-token "shown once" warning** — the result page says "save now — shown once" but the page can be reloaded by anyone with the same browser session. Should we hash+store-and-redirect, or change the wording to match reality?
5. **Mobile keyboard hint** — the PIN input has `inputmode="numeric"` ✅ but no `autocomplete="off"` on the IMAP password (the field has `autocomplete="new-password"` which is correct).
