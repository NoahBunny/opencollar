# Vault Mode — Zero-Knowledge Mesh Server

**Status:** Phases A–E + P6.5 (multi-signer) + P7 (Syncthing transport) SHIPPED, 2026-04-10
**Author:** Lion's Share + Collar maintainers
**Replaces:** plaintext orders relay in `focuslock-mail.py` (removed)

## 1. Goal

Make the relay server (`focuslock-mail.py`) a **dumb encrypted blob store**. The server should be able to operate a multi-tenant mesh without ever reading order contents, sub state, paywall amounts, geofence coordinates, or messages. Today it stores everything in plaintext (`/run/focuslock/orders.json`, plus the per-mesh metadata in `/run/focuslock/meshes/{mesh_id}.json`), which means a server admin or anyone with shell access on the relay host can read every order and message in every mesh.

The goal is *not* anonymity from the server (it still sees `mesh_id`, version numbers, ciphertext sizes, and timing). The goal is **content confidentiality**: the server can route and durably store, but it cannot interpret.

## 2. Threat model

| Adversary | Can do | Cannot do |
|---|---|---|
| **Public relay admin** (hosted Bunny Dev) | Read mesh_id, version, signature bytes, ciphertext sizes, timing, node IDs. Delete blobs (DoS). Refuse appends (DoS). | Read order contents. Forge orders (no Lion privkey on server — P6.5). Admin API is disabled for non-operator meshes. |
| **Homelab relay admin** (self-hosted, own mesh) | Above, plus: sign admin orders on behalf of Lion via the relay keypair (same trust domain — the operator IS the Lion, so this is expected). | Forge orders in *other* meshes (the relay is only a registered node in the operator's mesh). |
| Server admin running a different mesh on the same instance | See another tenant's blob count, sizes, timing | Cross-mesh correlation beyond what `mesh_id` already exposes. Read content. Sign orders for tenants (relay is only registered in the operator mesh). |
| Network attacker on the wire | Observe TLS metadata | Decrypt — protected by TLS *and* the vault layer below it |
| Malicious bunny node (compromised slave) | Read its own mesh contents (necessary — it must apply orders) | Forge orders to other bunny nodes (no Lion privkey). Promote itself to a controller. Take over another node's identity via `register-node-request` (key rotation requires Lion approval since 2026-04-10). |
| LAN peer without credentials | Observe plaintext gossip traffic (if LAN is hostile) — but orders are RSA-signed so integrity is preserved | POST to `/mesh/order` without a valid PIN or Lion signature (P1 fix, 2026-04-10). Push unsigned orders via `/mesh/sync` when `lion_pubkey` is configured (P1 fix, 2026-04-10). |
| Browser-based attacker against web UI | — | Run scripts via XSS (mesh-controlled strings are HTML-escaped, P2 fix 2026-04-10). Frame the web UI for clickjacking (`X-Frame-Options: DENY`). Read admin endpoints cross-origin (CORS scoped out of `/admin/*`). |
| Web session stealer (post-approval token leak) | Act as admin for at most 8 hours | Use the stolen token past TTL. Impersonate Lion for new approvals (a fresh session needs Lion's RSA signature). Access the master `ADMIN_TOKEN` (web UI only ever sees scoped session tokens since 2026-04-10). |
| Lost Lion phone | — | Continue locking. Recovery is via Release Forever or factory reset, same as today. |

**Out of scope for v1:**
- Forward secrecy / per-message keys (defer to v2 — Double Ratchet over the same transport)
- Hiding `mesh_id` from the server (a future Tor / Oxen routing layer)
- Anti-traffic-analysis padding

## 3. Cryptographic primitives

Reuse what `E2EEHelper.java` already implements in Lion's Share — no new crypto code on the controller side.

| Primitive | Choice | Why |
|---|---|---|
| Asymmetric encryption | RSA-2048 / OAEP-SHA256 | Already used in `E2EEHelper` (currently `RSA/ECB/PKCS1Padding`; **upgrade to OAEP** in v1 — see §10) |
| Symmetric encryption | AES-256-GCM | Already used in `E2EEHelper` |
| Signatures | RSA-PKCS1v15-SHA256 | Already used by `focuslock_mesh.py:sign_orders` / `verify_signature`. Out of an abundance of caution we'll move to RSA-PSS in v2; PKCS1v15 is fine for v1 since the signed payload is canonical-JSON of an already-encrypted blob. |
| Canonical encoding | `json.dumps(sort_keys=True, separators=(",",":"))` | Already used by `canonical_json()` in `focuslock_mesh.py:40` |
| Mesh ID derivation | `base64url(SHA256(lion_pubkey_DER))[:16]` | New — see §4 |

## 4. Mesh ID derivation

Currently `mesh_id` is a random 12-char string generated by the server (`focuslock-mail.py:_mesh_accounts.create`). In vault mode, the mesh_id is derived deterministically from the Lion's pubkey:

```
mesh_id = base64url_unpadded(SHA256(lion_pubkey_DER))[:16]
```

This means:
1. The mesh_id is reproducible — if Lion reinstalls Lion's Share with the same keypair (e.g., from a backup), the mesh_id is the same. No need to re-pair every bunny.
2. The server cannot generate a "fake" mesh_id and trick a bunny — the bunny verifies on first sync that `mesh_id == hash(lion_pubkey)`.
3. There is no auth_token to leak — Lion authenticates by *signing* every append with `lion_privkey`, and the server verifies against the `lion_pubkey` it stores under that mesh_id. (auth_token still exists for legacy mode, see §9.)

## 5. The vault envelope

The unit of storage in the vault is a **VaultBlob**. Each blob represents one snapshot of the orders document at a given version. The format:

```json
{
  "mesh_id":   "FxxJ19KEM_m7AB12",
  "version":   42,
  "created_at": 1775511776668,
  "slots": {
    "<sha256(node_pubkey)[:12]>": {
      "encrypted_key": "<base64 RSA-OAEP(AES key, node_pubkey)>",
      "iv":            "<base64 12 bytes>"
    },
    "<sha256(other_node_pubkey)[:12]>": { "encrypted_key": "...", "iv": "..." }
  },
  "ciphertext": "<base64 AES-256-GCM(canonical_json(orders), shared_aes_key)>",
  "signature":  "<base64 RSA-PKCS1v15-SHA256( canonical_json({mesh_id, version, created_at, slots, ciphertext}) )>"
}
```

**Why "slots":** the mesh has multiple bunny nodes (Pixel, BunnyTasker, desktop collars), each with its own RSA keypair. The Lion encrypts the AES key once for *each* registered node pubkey. This is the standard "envelope per recipient" pattern. The shared `ciphertext` is encrypted only once, so the blob grows linearly in node count for the small slot overhead (~300 bytes/slot) but the order body is paid once.

**Slot lookup:** when a node receives a blob, it computes `sha256(its_own_pubkey)[:12]` and pulls its slot out. If no slot matches, the blob is not for it (rejected).

**Signature scope:** the Lion's signature covers everything *except* the signature field itself. The server can verify the signature is valid and authored by the registered `lion_pubkey` for this `mesh_id`, **but cannot decrypt the ciphertext**. The signature gives the server enough to reject spam appends from non-Lions.

## 6. Server endpoints (`focuslock-mail.py` additions)

All vault endpoints live alongside the existing `/api/mesh/...` endpoints. They are gated by the `vault_mode` config flag (see §9).

### 6.1 `POST /vault/{mesh_id}/append`

Body: a single `VaultBlob` (see §5).

Server:
1. Look up `lion_pubkey` for `mesh_id`. If unknown, **404**.
2. Verify `signature` against `lion_pubkey`. If invalid, **403**.
3. Verify `blob.version > current_version[mesh_id]`. If not, **409 Conflict**.
4. Verify `blob.mesh_id` matches the URL `mesh_id`. If not, **400**.
5. Persist to `/run/focuslock/vaults/{mesh_id}/{version:08d}.json`.
6. Update `current_version[mesh_id] = blob.version`.
7. Respond `{"ok": true, "version": <version>}`.

The server **never decodes the ciphertext**. It treats it as opaque bytes.

**Retention:** the latest blob per mesh is always retained. Older blobs are GC'd after `vault_retention_days` (default 7) — clients should sync at least that often. This bounds disk usage.

### 6.2 `GET /vault/{mesh_id}/since/{version}`

Returns all blobs with `version > {version}`, in ascending order:

```json
{
  "current_version": 42,
  "blobs": [
    { /* VaultBlob */ },
    ...
  ]
}
```

No auth required for reads — the blobs are ciphertext, and the server already exposes mesh_id via the URL. A bunny that doesn't have the right slot key gets nothing useful from a blob anyway. (We could require a per-mesh read token in v2.)

### 6.3 `POST /vault/{mesh_id}/register-node`

Body:
```json
{
  "node_id": "pixel-10",
  "node_type": "phone",
  "node_pubkey": "<base64 X.509 SubjectPublicKeyInfo>",
  "signature": "<Lion signs {mesh_id, node_id, node_pubkey} with lion_privkey>"
}
```

Server validates the Lion signature, then stores `node_pubkey` in the per-mesh node registry. New nodes added here will receive a slot in subsequent VaultBlobs (Lion's Share polls this list before each append).

### 6.4 `GET /vault/{mesh_id}/nodes`

Returns the list of registered nodes (`{node_id, node_type, node_pubkey, registered_at}`) so Lion's Share can encrypt to all of them. Anyone can read this — pubkeys are not secrets.

## 7. Client flows

### 7.1 Lion's Share (controller) — sending an order

```
1. Build orders dict (existing path — no change)
2. orders_json = canonical_json(orders)
3. aes_key = randbytes(32); iv = randbytes(12)
4. ciphertext = AES_GCM_encrypt(orders_json, aes_key, iv)
5. nodes = GET /vault/{mesh_id}/nodes
6. for each node in nodes:
       slot[sha256(node.pubkey)[:12]] = {
         encrypted_key: RSA_OAEP_encrypt(aes_key, node.pubkey),
         iv: iv,
       }
7. blob = {mesh_id, version: current+1, created_at: now_ms, slots, ciphertext}
8. blob.signature = RSA_PKCS1v15_sign(canonical_json(blob), lion_privkey)
9. POST /vault/{mesh_id}/append  body=blob
10. on 409 Conflict: GET /vault/{mesh_id}/since/{local} to learn current version, retry from step 6 with current+1
```

The encryption work happens in `E2EEHelper.encrypt(canonical_json, ...)` — already implemented. The slot fan-out is new but trivial (5–10 lines in `MainActivity.java`).

### 7.2 Collar (slave) — receiving orders

The Collar's existing gossip loop (`ControlService.meshGossip()`) gets a small change: instead of `POST /api/mesh/{mesh_id}/sync` returning a plaintext orders dict, it does:

```
1. blobs = GET /vault/{mesh_id}/since/{local_version}
2. for each blob in blobs (ascending version):
     a. verify blob.signature against stored lion_pubkey  → reject on fail
     b. my_slot = blob.slots[sha256(my_pubkey)[:12]]      → reject if missing
     c. aes_key = RSA_OAEP_decrypt(my_slot.encrypted_key, my_privkey)
     d. orders_json = AES_GCM_decrypt(blob.ciphertext, aes_key, my_slot.iv)
     e. orders = json.loads(orders_json)
     f. apply_orders(orders)  (existing path)
     g. local_version = blob.version
3. continue on next gossip tick
```

**New code on the slave side:** an Android counterpart of `E2EEHelper` (call it `VaultCrypto.java`) — RSA-OAEP decrypt + AES-GCM decrypt + RSA-PKCS1v15 verify. ~80 LOC, mirrors `E2EEHelper`. Plus the gossip loop change in `ControlService.java`.

### 7.3 BunnyTasker (companion) — same as Collar

`BunnyTasker.MainActivity.meshSync()` does the same dance. It can either:
- Have its own keypair and slot, OR
- Share the Collar's keypair (BunnyTasker reads it via `Settings.Global` since they cohabit)

**Recommendation:** share the Collar's keypair. Less slot bloat, simpler key management. The companion doesn't need a separate identity from the slave — they're the same physical device.

### 7.4 Desktop collars

Each desktop has its own RSA keypair persisted at `~/.config/focuslock/node_privkey.pem`. On install, `focuslock-desktop.py` generates the keypair and registers via §6.3. After that, it appears in `GET /vault/{mesh_id}/nodes` and gets a slot in every subsequent blob.

A new `vault.py` shared module (alongside `focuslock_mesh.py`) implements the encrypt/decrypt/verify primitives for the Python side. It will be used by `focuslock-desktop.py`, `focuslock-desktop-win.py`, and any future Python clients.

## 8. What moves client-side

Today the server does work that requires reading order contents:
- **Payment detection:** IMAP polling of bank emails, then writing `paywall` reductions into orders (`focuslock-mail.py` payment listener)
- **Photo verification:** uploading photos to Ollama for vision LLM scoring
- **Subscription tracking:** auto-charging weekly, computing `sub_due`
- **Compound interest:** hourly paywall accrual

In vault mode, the server cannot read or write the orders directly — so these must move to a place that *can*. Options per workload:

| Workload | Vault-mode home | Notes |
|---|---|---|
| Payment detection | Lion's Share | The Lion's phone polls IMAP via the existing `focuslock-mail.py` IMAP creds, OR Lion confirms payments manually in-app. Manual is fine for v1. |
| Photo verification | Lion's Share + homelab | The slave uploads encrypted photo to the server's blob store; Lion's Share fetches+decrypts+sends to a *user-controlled* Ollama (could still be the homelab, but as a user-side resource) |
| Subscription auto-charge | Lion's Share | Schedule a daily check on the controller; Lion confirms or rejects |
| Compound interest | Slave (Collar) | The Collar already runs continuously and can compute interest locally on `paywall_original` + elapsed_hours. No server involvement needed. |

**Migration path for these workloads:** none of them block v1 vault mode. The server will keep doing them in *legacy mode* meshes; vault-mode meshes opt out and do them client-side. Both modes coexist on the same server instance.

### Phase F — Multi-signer + Zero-knowledge relay (SHIPPED 2026-04-10, P6.5)

The server previously held Lion's private key to sign admin-originated vault blobs (`_admin_order_to_vault_blob()`). This broke the zero-knowledge claim: a hosted relay operator could read nothing but COULD forge orders on behalf of Lion.

**Fix:** the server now generates its own RSA-2048 keypair at `~/.config/focuslock/relay_{priv,pub}key.pem` on first run. It auto-registers as a vault node with `node_type: "relay"`, `node_id: "relay"` for the operator's mesh (gated by `OPERATOR_MESH_ID`). Admin orders are signed with the relay keypair. Lion's private key is no longer present on the server.

**Trust tiers:**
- **Public relay (Bunny Dev):** no `OPERATOR_MESH_ID`, no admin API, no signing keys for any mesh. Pure ciphertext blob store. Zero-knowledge by construction.
- **Homelab / self-hosted:** `OPERATOR_MESH_ID` set, admin API available for that mesh only. Server signs admin orders with relay keypair. Clients accept relay-signed blobs because the relay is a registered node in the approved list.

**Multi-signer verification:** the previous verification logic only accepted blobs signed by Lion or self. Now clients fetch the approved-nodes list from `/vault/{mesh_id}/nodes` and verify against any registered pubkey:

- **Server** (`_verify_blob_two_writer`): Lion pubkey → iterate all registered node pubkeys
- **Slave** (`ControlService.vaultSync`): Lion → self → any approved node (`fetchApprovedNodes()` with 30-minute cache, lazy refresh on miss, rate-limited to once per poll)
- **Controller** (`MainActivity.vaultPollLoop`): Lion-signed → orders. Relay-signed → orders (node_type "relay"). Other node-signed → runtime state pushes. Splits `relayPubkeys` from `slavePubkeys`.
- **Desktop collars** (`_vault_verify_any_signer`): Lion → any approved node, lazy refresh, rate-limited

**Key rotation security:** previously `register-node-request` auto-approved if `node_id` matched an existing active node. This was an unauthenticated takeover vector — an attacker knowing a mesh_id + node_id could replace any node's pubkey. **Removed in 2026-04-10.** Key rotation now requires Lion approval via the pending queue.

### Phase G — Transport abstraction + Syncthing P2P (SHIPPED 2026-04-10, P7)

The HTTP relay is now one of multiple pluggable transports. New module `shared/focuslock_transport.py` defines:

```python
class VaultTransport:
    def since(mesh_id, version) -> (blobs, current_version)
    def append(mesh_id, blob) -> (version, error)
    def nodes(mesh_id) -> list[dict]
    def register_node(mesh_id, payload) -> dict
```

Two implementations:
- **`HttpVaultTransport`** — wraps the current HTTP relay. 10 MB response cap to prevent OOM from a malicious server.
- **`SyncthingVaultTransport`** — directory-based P2P transport. Blobs are written as files to a Syncthing-shared folder, same format as the server's `VaultStore` uses (`{vault_dir}/{mesh_id}/blobs/{version:08d}.json`). Clients read by scanning the directory.

**Config:**
```json
{
  "vault_transport": "http",  // or "syncthing"
  "syncthing_vault_dir": "~/Syncthing/focuslock-vault/"
}
```

**Why Syncthing:** it sells the hosted Bunny Dev relay by giving users a credible exit lane. Switch `vault_transport` to `syncthing`, point Lion's Share and the collars at the same Syncthing folder, and the mesh works fully P2P with zero VPS, zero relay, zero ongoing cost. No lock-in.

**Security hardening on the Syncthing transport:**
- Mesh ID and node ID are validated with `_safe_id()` (alphanumeric, max 64 chars)
- All paths go through `_safe_path()` which uses `os.path.realpath()` to verify the resolved path stays inside `vault_dir` (defeats path traversal and symlink escape)
- `os.path.islink()` checks on every file read (defeats runtime symlink swaps)
- Version numbers parsed from filenames are bounded `0 < v ≤ 2³¹−1` to prevent a malicious peer poisoning `current_version` with `99999999.json`

**Android support deferred** — Syncthing-Fork + Storage Access Framework integration is out of scope for this phase. Desktop collars support Syncthing immediately; Android collars remain HTTP-only until a future phase.

## 10. Crypto upgrades during migration

The current `E2EEHelper.java` uses `RSA/ECB/PKCS1Padding` — that's RSA-PKCS1v15 encryption, which has known padding-oracle attack risks. **For vault mode, switch to RSA-OAEP with SHA-256 main hash and MGF1-SHA1:**

- Java: `Cipher.getInstance("RSA/ECB/OAEPWithSHA-256AndMGF1Padding")` with `OAEPParameterSpec("SHA-256", "MGF1", MGF1ParameterSpec.SHA1, PSource.PSpecified.DEFAULT)`
- Python: `cryptography.hazmat.primitives.asymmetric.padding.OAEP(mgf=MGF1(SHA1()), algorithm=SHA256(), label=None)`

**Why MGF1-SHA1 instead of MGF1-SHA256** (as of slave v57, controller v61): AndroidKeyStore's RSA-OAEP provider internally calls MGF1 with SHA-1 regardless of what `OAEPParameterSpec` declares — it only honors the parameter if SHA-1 is authorized in `setDigests()`. To make hardware-backed node keys decryptable by software-generated ciphertext, the whole protocol uses MGF1-SHA1. This has no practical security cost: MGF1 uses the hash as a PRF, where SHA-1's collision weaknesses are not relevant.

Decrypt paths have a fallback to MGF1-SHA256 for blobs produced by pre-v57 peers during the transition window. This fallback will be removed once all peers have upgraded.

**Signature algorithm:** keep RSA-PKCS1v15-SHA256 in v1 (matches what the existing `focuslock_mesh.py` already does). No MGF1 involved in PKCS1v15 signatures, so the landmine #1 issue does not apply to signatures. PSS migration is a v2 concern — both interop fine and PKCS1v15 signatures aren't subject to the same padding-oracle issues that PKCS1v15 *encryption* is.

## 11. Open questions and gotchas

**Q1. Chicken-and-egg on first slave registration.**
The slave wants to register its pubkey, but the server requires a Lion signature on `register-node` calls (to prevent rogue registration). On first install, the slave has no Lion to sign for it. Resolution: Lion's Share polls `GET /vault/{mesh_id}/nodes-pending` (a small extension to §6.3) for unsigned registrations and approves them via a UI prompt. This mirrors the existing "Pair Direct (LAN)" flow's pubkey-exchange step.

**Q2. Lost slave keypair.**
If a slave reinstalls and loses its keypair, it can no longer decrypt past blobs. Resolution: register a new keypair, Lion's Share appends a fresh blob with the new slot, slave catches up from the new version onward. Older blobs become unreadable to that node — this is actually a feature (forward secrecy lite).

**Q3. Multi-bunny meshes (P0 from the roadmap).**
Multi-bunny support is orthogonal to vault mode. Each bunny is its own *mesh* (different `mesh_id`, different lion_pubkey-derived ID), so vault mode adds no new complexity here. The "multi-bunny on one Lion's Share" feature on the roadmap is about the controller UI managing multiple meshes, not about putting multiple bunnies in one mesh.

**Q4. Slot count growth.**
Each desktop adds ~300 bytes per append. A mesh with 1 phone + 5 desktops = ~1.8 KB of slot overhead per blob, plus the encrypted body (typically <1 KB). Even at 1 blob/minute that's ~4 MB/day before GC, well within disk budget.

**Q5. Server ciphertext tampering.**
A malicious server admin could edit the ciphertext. The Lion's signature covers the ciphertext, so any edit invalidates the signature and the slave rejects the blob. Net effect of tampering: DoS, which the admin can already cause by deleting blobs. No new threat.

**Q6. GC race.**
If a slave is offline for >`vault_retention_days` and the latest blob is GC'd of intermediate blobs, the slave catches up to the latest version on next sync — but loses any "intermediate" state changes. Since orders are *idempotent snapshots* (not deltas), this is fine: applying the latest blob is equivalent to applying every intermediate one.
